Listor

Från enkla inloggningar till avancerade nyttoprogram så används formulär (forms). CMS:er sköts via formulär. Bloggar, Facebook och Twitter får sina data via formulär.
I och med HTML5 och CSS 3, så finns det numera så många olika element, attribut, egenskaper och selektorer att välja på att man lätt går vilse. Här finns varken plats eller någon nytta med att gå igenom allt i detalj. Vi satser på grunderna och avslutar med ett enkelt exempel.

Formulär, http och https

  1. Formulär
  2. Bearbeta
  3. Visa Resultat

Listan ovan illustrerar formulär i dess enklaste form. Webbläsaren, punkt 1, visar ett formulär som skickas (submit) till servern, punkt 2. Där bearbetas den inmatade datan med någon slags server-side skript och ett resultat visas, punkt 3, oftast i form av en bekräftelse eller ett felmeddelande.
Det senare bör upprepa formuläret med de gamla värdena, så att man bara behöver ändra det som blivit fel.

Data skickas med http. Om känslig data skickas, såsom kreditkortsinformation, så måste förbindelsen vara krypterad med SSL/TLS. Det finns två sätt att skicka datata: GET och POST.
tab Metoden GET visar vad som skickas i sidans URL och olika webbläsare har begränsningar för hur lång en URL kan vara, så denna metod kan inte användas för stora mängder data. URL:er bör även vara läsliga och kunna klistras in i mejl eller på sociala nätverk, vilket begränsar mängden ytterligare. För sökformulä, så är GET bra, nästan aldrig för andra sorters formulär. För lösenord (inloggning, registrering) är det en dödssynd att använda GET, eftersom det kommer synas i klartext i adressfältet, oavsett alla andra åtgärder du vidtagit för att skydda det.
Metoden POST skickar datan som ett eget dokument (message body) istället för en del av http-huvudet.

Elementet

Formulär omges av start- och sluttaggen för elementet <form>.

<form action="login.php" method="post">
  ...
</form>
Formuläret har alltid två egenskaper, action och method, även om de inte skrivs ut som attribut. Med attributet action kan man tala om vart formuläret skickas, d.v.s på vilken URL serverns skript återfinns. Med attributet method väljer du mellan GET och POST. Som attributvärde spelar det ingen roll om du använder stora eller små bokstäver.
Om du inte anger action, så skickas formulärets data till sidans egen adress.

Synkronisering

För dina första formulär så har antagligen någon annan gjort det skript som körs på servern. Att göra säkra skript är nämligen inte helt enkelt. Det finns många slags faror.
Serverskripten förväntar sig Data enligt mönstret nyckel - värde (key - value). Nyckeln kallas här också namn (name) och om du använder ett färdigt skript är det viktigt att du läser instruktionerna, så att du väljer rätt namn.
Skripten kan också producera felmeddelanden eller annan information som ska visas för användarna. Hur de infogas i HTML-koden är en annan sak du behöver kontrollera.

Inloggningsformulär

Ett mycket enkelt inloggningsformulär kan kodas så här:

<form action="login.php" method="post">
  <p>
    <label for="name">Användarnamn: </label>
    <input type="text" name="uName" id="name" value="användarnamn" required/>
  </p>
  <p>
    <label for="pass">Lösenord: </label>
    <input type="password" name="uPass" id="pass" value="password" required />
  </p>
  <p>
    <input type="submit" value="Logga in">
  </p>
</form>

Name, value och id

I koden ovan förekommer tre grundläggande attribut.

  • Attributet name är främst till för servern. Det är nyckeln som vi talat om tidigare. Värder är vanligtvis unikt, utom för radioknappar och checkbox-rutor (se nedan), där det används för gruppering.
  • Attributet value används ovan för att styra texten på submitknappen, men är annars ett startförslag som visas i textrutor. för knappar av olika slag är value själva värdet som skickas.
  • Attributet id är enbart till för användning på klienten. Det påverkar inte skickad data. Ovan länkas förklarande text i <label> samman med motsvarande input-element via attributen for och id. Id används inte för gruppering, eftersom varje id måste vara unikt.
Om du vill ge ett tips till användaren som inte också är ett föreslaget värde, så använd attributet placeholder, hellre än value.
